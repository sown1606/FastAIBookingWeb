## Why previous fixes did not resolve live speech

Previous validation mainly exercised already-transcribed text. It did not prove speech recognition quality over the Vietnam-to-US PSTN/ViberOut path.

The deployed Lex locale still had polluted Full Set slot synonyms, while the live bot locale speech settings reported null unified speech settings and null speech detection sensitivity. Runtime hints generated by Lambda could only affect later turns; the first caller utterance was still being handled by the Connect Lex block before Lambda could provide dynamic runtime hints.

The Lambda debug payload also treated NLU confidence as if it were speech transcription confidence, creating misleading `confidence: 1` diagnostics for wrong transcripts such as `sunset is it food`.

## Root causes

ASR diagnostics:

- Lambda used `interpretations[].nluConfidence.score` as a fallback for speech confidence.
- That made semantic/NLU confidence look like `transcriptionConfidence: 1`.
- API consumed ASR diagnostic alternatives without clearly separating transcription confidence and NLU confidence.

Lex:

- Full Set slot synonyms had malformed ASR outputs such as `fun fact`, `food set`, `phone set`, `pool set`, and `cool set`.
- Staff slot/source aliases included malformed any-staff variants such as `any stuff`.
- Runtime hints were not proven active for the first utterance.

Lambda/API state:

- Malformed service phrases could commit Full Set directly.
- Malformed first-available phrases could commit Any staff directly.
- A staff turn could previously mutate a trusted time when incidental time-like words appeared.

Connect:

- Normal Lex error/no-match recovery could enter `retry_after_initial_error`, play an extra recovery message, and make the conversation feel restarted.
- Connect error metadata was not captured clearly enough for turn-level diagnosis.

Latency:

- The available logs showed Lambda/API processing was usually smaller than perceived caller silence.
- Full end-of-speech-to-prompt-start timing could not be measured without real audio/contact event timing.

Malformed phrases may now only create a low-trust proposed value in the correct active slot context, requiring confirmation.

## Slot mutation policy

One shared policy is now applied in Lambda and API normalization:

- A slot can be updated freely only when it is the active `lastAskedSlot`.
- A previously trusted slot can be replaced by an explicit correction such as `change`, `instead`, `make it`, or `not 3`.
- A clearly structured complete booking request can provide multiple grounded slots.
- Bare or ambiguous staff-turn text cannot overwrite `requestedTime`.
- Service-slot answers cannot overwrite trusted staff/time with unrelated noise.
- Proposed Full Set or first-available values are stored as `proposedServiceName` / `proposedStaffPreference` with confirmation flags and are not committed until affirmative confirmation.

## First-turn hint implementation

Source Connect flow now routes the first Lex interaction into structured booking context with `lastAskedSlot=serviceName`, `initialVoiceBookingContext=true`, and audio timeout profile attributes.

Slot-specific hints remain a limitation for the first utterance: no production evidence showed Lambda-built runtime hints present before the first caller audio reaches Lex. A larger architecture change, such as pre-Lex runtime session setup or a dedicated ASR layer, is required if first-utterance runtime hints must be guaranteed.

## Connect flow before/after

Before:

- Primary Lex block handled first utterance.
- Error path set recovery stage and played a separate recovery message.
- Recovery could re-enter Lex with `retry_after_initial_error`, causing repeated welcome/recovery behavior.

After source change:

- Primary Lex block starts in structured booking context.
- Audio timeout profile and speech context attributes are set at the Lex block.
- Error attributes capture `connectErrorCode`.
- Recovery paths route directly back into Lex without an extra overlapping MessageParticipant prompt.
- Operator transfer remains explicit-only through voice request or DTMF 0.
- Nonterminal branches continue the conversation rather than disconnecting.
